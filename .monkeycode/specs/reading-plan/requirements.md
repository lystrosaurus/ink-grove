## 2. 产品愿景

基于 gtool 阅读规划系统的 200+ 本经典著作书单作为初始基础，在 ink-grove 知识管理仓库中建立系统化的阅读工作流和持续扩展机制，通过规范化的书单管理、阅读记录和知识沉淀，实现高质量的深度阅读与知识积累，并支持从多来源持续补充新书。

---

## 3. 范围

### 3.1 包含内容

- **全量导入 gtool 书单**：将 gtool 仓库的 12 大领域 200+ 本经典著作完整导入到 ink-grove 的 book-list/to-read/ 目录
- **持续书单扩展**：支持从其他来源（如推荐书单、豆瓣、知乎、书籍等）持续补充新书到阅读清单
- 在 ink-grove 的 book-list 目录中管理书单条目（待读/在读/已读）
- 每次阅读一本书时，在 reading-log 目录创建阅读记录
- 读完一本书或一个章节后，在 knowledge-notes 目录沉淀知识笔记
- 定义高质量文档的格式标准、内容深度和命名规范

### 3.2 不包含内容

- gtool 仓库本身的代码开发或修改
- 自动化工具或脚本开发
- 与其他阅读平台的集成

## 5. 需求描述

### 5.1 业务需求

#### BR-001：阅读清单整理

**EARS 表述：**
- **简易主动语句**：系统应支持全量导入 gtool 仓库的 12 大领域 200+ 本经典著作，并支持从多来源持续补充新书
- **事件驱动**：当用户启动阅读计划时，系统应将 gtool 的完整书单批量导入到 book-list/to-read/ 目录
- **状态驱动**：当书籍被添加到阅读清单时，系统应将其标记为"待读"状态并放入 book-list/to-read/ 目录
- **异常处理**：如果 gtool 仓库不可访问，系统应支持用户手动输入书籍信息；如果新增书籍与现有书籍重复，系统应提示用户去重

**验收标准：**
1. gtool 书单 100% 完整导入，不遗漏任何书籍
2. 每本书籍有完整的元数据（书名、作者、出版社、分类、领域、来源）
3. 支持从其他来源（豆瓣、知乎、推荐书单等）添加新书
4. 新增书籍自动标记来源和添加日期
5. 书单按 gtool 的 12 大领域分类组织，支持新增分类

#### BR-002：单书阅读工作流

**EARS 表述：**
- **简易主动语句**：系统应支持每次只阅读一本书籍的深度阅读模式
- **事件驱动**：当用户开始阅读一本书时，系统应将书籍状态从"待读"改为"在读"并移动到 book-list/reading/ 目录
- **状态驱动**：当用户完成一本书的阅读时，系统应将书籍状态改为"已读"并移动到 book-list/done/ 目录
- **异常处理**：如果用户中途暂停阅读，系统应保持书籍在 book-list/reading/ 目录并记录暂停原因

**验收标准：**
1. 同一时间只有一本书处于"在读"状态
2. 书籍状态变更有明确的时间记录
3. 支持查看阅读历史统计（已读数量、阅读时长等）

#### BR-003：阅读记录创建

**EARS 表述：**
- **简易主动语句**：系统应支持每次阅读后在 reading-log 目录创建阅读记录
- **事件驱动**：当用户完成一次阅读会话时，系统应引导用户创建阅读记录
- **状态驱动**：当阅读记录创建时，系统应使用 templates/reading-log.md 模板
- **异常处理**：如果用户跳过阅读记录创建，系统应在下次提醒补全

**验收标准：**
1. 阅读记录文件命名符合 `YYYY-MM-DD-书名-简述.md` 格式
2. 记录包含日期、页码范围、摘录和感想
3. 支持按时间线查看某本书的完整阅读记录

#### BR-004：知识笔记沉淀

**EARS 表述：**
- **简易主动语句**：系统应支持在读完一本书或一个章节后在 knowledge-notes 目录沉淀知识笔记
- **事件驱动**：当用户完成一本书或一个重要章节的阅读时，系统应引导用户创建知识笔记
- **状态驱动**：当知识笔记创建时，系统应使用 templates/knowledge-note.md 模板
- **异常处理**：如果知识笔记内容不完整，系统应标记为"草稿"状态

**验收标准：**
1. 知识笔记包含核心要点、深入理解、实践启发
2. 笔记之间建立跨书籍的知识关联
3. 支持按主题检索相关知识笔记

### 5.2 功能需求

#### FR-001：书单条目管理

**EARS 表述：**
- **简易主动语句**：系统应支持在 book-list 目录中添加、更新和移动书单条目
- **事件驱动**：当用户添加新书时，系统应使用 templates/book-entry.md 模板创建书单条目
- **状态驱动**：当书籍状态变更时，系统应将文件移动到对应子目录（to-read/reading/done）
- **异常处理**：如果文件名不符合命名规范，系统应提示用户修正

**验收标准：**
1. 书单条目文件命名符合 `书名 - 作者.md` 格式
2. 条目包含完整元数据（书名、作者、出版社、页数、分类、状态、想读理由、评价）
3. 支持按状态、分类、领域筛选书单

#### FR-002：高质量文档标准

**EARS 表述：**
- **简易主动语句**：系统应确保所有文档符合高质量标准
- **事件驱动**：当用户创建或更新文档时，系统应验证文档是否符合标准
- **状态驱动**：当文档质量不达标时，系统应标记为"待完善"
- **异常处理**：如果文档格式错误，系统应提供修正建议

**验收标准：**
1. 所有文档使用标准 Markdown 格式，不使用 HTML 标签
2. 标题层级规范（一级、二级、三级标题正确使用）
3. 列表、引用、代码块等格式统一
4. 中文书写，技术术语保留英文原样
5. 文件命名不使用空格，统一使用连字符分隔

#### FR-003：阅读进度追踪

**EARS 表述：**
- **简易主动语句**：系统应支持追踪每本书的阅读进度
- **事件驱动**：当用户更新阅读记录时，系统应计算并显示当前进度百分比
- **状态驱动**：当进度达到 100% 时，系统应提示用户完成阅读并创建知识笔记
- **异常处理**：如果进度数据缺失，系统应显示"未知"状态

**验收标准：**
1. 进度以页码或章节数计算
2. 支持查看每本书的阅读时间线
3. 支持导出阅读进度报告

#### FR-004：知识网络构建

**EARS 表述：**
- **简易主动语句**：系统应支持在知识笔记之间建立关联
- **事件驱动**：当用户创建知识笔记时，系统应引导用户填写"延伸阅读"字段
- **状态驱动**：当多篇笔记关联同一主题时，系统应形成知识网络
- **异常处理**：如果关联的笔记不存在，系统应标记为"待创建"

**验收标准：**
1. 知识笔记包含"关联知识点"字段
2. 支持按主题查看相关知识笔记集合
3. 支持可视化知识网络（可选）

### 5.3 非功能需求

#### NFR-001：文档一致性

**EARS 表述：**
- **简易主动语句**：所有文档应保持一致的格式和风格
- **事件驱动**：当新文档创建时，系统应强制使用对应模板
- **状态驱动**：当文档风格不符合规范时，系统应提示修正
- **异常处理**：如果模板文件缺失，系统应提供默认内容

**验收标准：**
1. 所有书单条目使用相同的字段结构
2. 所有阅读记录使用相同的记录格式
3. 所有知识笔记使用相同的章节组织

#### NFR-002：可维护性

**EARS 表述：**
- **简易主动语句**：文档应易于维护和更新
- **事件驱动**：当文档内容变更时，系统应记录修改日期
- **状态驱动**：当文档长期未更新时，系统应标记为"待审查"
- **异常处理**：如果文档内容冲突，系统应提示用户核实

**验收标准：**
1. 文档包含最后更新日期
2. 支持文档版本追溯（通过 Git 历史）
3. 支持批量更新和重构

#### NFR-003：可扩展性

**EARS 表述：**
- **简易主动语句**：系统应支持未来扩展新的文档类型
- **事件驱动**：当新增文档类型时，系统应允许创建新模板
- **状态驱动**：当目录结构变更时，系统应保持向后兼容
- **异常处理**：如果扩展导致冲突，系统应回滚变更

**验收标准：**
1. templates 目录支持添加新模板
2. 目录结构允许新增子目录
3. 现有文档不受扩展影响

#### NFR-004：版权合规

**EARS 表述：**
- **简易主动语句**：文档内容应符合版权合理使用原则
- **事件驱动**：当用户添加摘录内容时，系统应提示版权注意事项
- **状态驱动**：当摘录内容超过合理范围时，系统应发出警告
- **异常处理**：如果收到版权投诉，系统应支持下架相关内容

**验收标准：**
1. 摘录内容控制在合理引用范围内
2. 所有摘录标注来源（书名、作者、页码）
3. 不记录受版权保护的完整章节

## 6. 其他质量需求

### 6.1 文档质量

- **准确性**：所有书籍元数据（书名、作者、出版社）必须准确无误
- **完整性**：书单条目必须填写所有必填字段
- **一致性**：同一本书在不同文档中的表述必须一致
- **时效性**：阅读记录和知识笔记应及时创建，不超过阅读后 48 小时

### 6.2 内容深度

- **阅读记录**：每次阅读至少记录 3 条摘录或感想
- **知识笔记**：每本书至少沉淀 5 个核心知识点
- **关联深度**：每篇知识笔记至少关联 1 篇其他笔记或书籍

## 8. 约束条件

| 约束 | 说明 | 影响 |
|------|------|------|
| 纯 Markdown 格式 | 不使用 HTML 标签或其他富文本格式 | 限制表达能力但保证兼容性 |
| 中文书写 | 所有笔记正文使用中文 | 技术术语可保留英文 |
| 无自动化 | 不开发自动化脚本或工具 | 依赖人工操作 |
| 单书阅读 | 每次只阅读一本书 | 不支持并行阅读多本书 |

## 9. 验收标准

### 9.1 阅读清单验收

- [ ] gtool 书单 100% 完整导入（12 大领域 200+ 本）
- [ ] 每本书有完整的元数据（书名、作者、分类、来源、添加日期）
- [ ] 书籍按 12 大领域分类组织到 book-list/to-read/

- [ ] 建立书单扩展记录文档（记录新增书籍的来源和理由）

### 9.2 工作流验收

- [ ] 完成至少 1 本书的完整阅读工作流演示
- [ ] 创建书单条目、阅读记录、知识笔记各至少 1 份
- [ ] 所有文档符合命名规范和格式标准
- [ ] 书籍状态正确流转（待读→在读→已读）

### 9.3 文档质量验收

- [ ] 所有文档通过 Markdown 语法校验
- [ ] 所有文档符合模板要求
- [ ] 知识笔记包含至少 5 个核心知识点
