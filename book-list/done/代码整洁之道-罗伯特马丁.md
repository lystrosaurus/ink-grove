# 代码整洁之道

状态：已读

作者：罗伯特 - 马丁（Uncle Bob）

一句话评价：程序员的"写作规范"——好代码不是给机器跑的，是给人读的。写代码和写文章一样，清晰比炫技重要一万倍，这本书影响了整整一代程序员的代码审美观。

核心收获：
- **命名的力量**: 变量名、函数名、类名都要"自解释"——读到名字就知道它是什么、做什么。`calculatePay()` 比 `cp()` 好一万倍，`isUserEligibleForDiscount` 比 `flag` 好一万倍。命名是编程中最难也最重要的事
- **函数只做一件事**: 一个函数如果超过 20 行，它大概率在做不止一件事。函数应该像一篇文章的段落——有明确的主题句，有清晰的逻辑流，没有多余的东西
- **注释是代码的失败**: 这不是说永远不要写注释，而是说——如果你需要用注释来解释代码在做什么，说明你的代码写得不够清晰。好代码本身就是最好的文档
- **代码坏味道**: 过长函数、过长参数列表、重复代码、过度耦合、过深的嵌套——这些都是"坏味道"，闻到了就要重构。Martin Fowler 说得好："任何一个傻瓜都能写出机器能懂的代码，只有好程序员能写出人能懂的代码"
- **重构是持续过程**: 重构不是"有空再做"的技术债，是每写一段代码都要问"能不能更整洁"的习惯。童子军规则：离开营地时要比来时更干净
- **格式即纪律**: 代码的缩进、空行、变量声明顺序——这些看似"小事"，体现的是程序员的职业纪律。混乱的代码传递的信号是："我不在乎"
- **错误处理的艺术**: 不要用 null 返回值，不要用错误码，用异常。但异常应该只在"异常"情况下抛出——正常的业务逻辑不应该被 try-catch 包裹
- **测试代码也是代码**: 很多人把测试代码当成"二等公民"，但 Uncle Bob 说：测试代码的质量决定了生产代码的可维护性。测试要清晰、简洁、可读
